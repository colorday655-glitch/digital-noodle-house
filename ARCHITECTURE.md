# 无人面馆 - 技术架构方案

## 1. 技术栈推荐

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 前端 | Next.js + Vanilla CSS | 极简、无需复杂构建 SSR 友好 |
| 后端 | Next.js API Routes | 前后一体，快速上手 |
| 数据库 | SQLite（初期）/ PostgreSQL | 零配置、嵌入式、后期可迁移 |
| 实时通信 | SSE（Server-Sent Events） | 轻量、单项推送、适合后厨通知 |
| 部署 | Vercel / 树莓派本地部署 | 免费极简 |

---

## 2. 数据流向设计

```
┌─────────┐     ┌──────────┐     ┌───────────┐     ┌──────────┐
│ 用户点击 │────▶│ Next.js  │────▶│  SQLite   │     │  后厨屏  │
│ 按钮     │     │ API      │     │  订单库   │────▶│  SSE推送 │
└─────────┘     └──────────┘     └───────────┘     └──────────┘
```

### 核心 API 设计

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/orders` | POST | 创建订单，返回订单ID |
| `/api/orders/[id]` | GET | 查询订单状态 |
| `/api/kitchen/events` | GET | SSE，后厨实时接收新订单 |

### 订单状态流转
```
pending（待接单） → cooking（制作中） → ready（已出餐） → completed（已取餐）
```

## 3. 稳定性设计方案

### 3.1 并发处理

| 策略 | 实现方式 |
|------|----------|
| 请求限流 | 同一设备 30 秒内最多下单 1 次 |
| 队列缓冲 | 使用 Redis 队列（高峰期）或直接写入 SQLite（低峰） |
| 数据库隔离 | 订单表索引：order_id, status, created_at |

### 3.2 高可用设计

```
                    ┌─────────────┐
                    │  负载均衡   │
                    │ （Nginx）  │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
      ┌─────────┐   ┌─────────┐   ┌─────────┐
      │ 节点 1  │   │ 节点 2  │   │  节点 3 │
      └─────────┘   └─────────┘   └─────────┘
           │               │               │
           └───────────────┴───────────────┘
                           │
                    ┌──────┴──────┐
                    │  共享存储    │
                    │  SQLite/NFS │
                    └─────────────┘
```

### 3.3 故障应对

| 场景 | 方案 |
|------|------|
| 数据库宕机 | 内存缓存订单，恢复后批量写入 |
| 后厨屏断网 | 轮询 `/api/orders?status=pending` 每 10 秒 |
| 超时红包 | 后端定时任务扫描超时订单，自动发放补偿券 |

### 3.4 压测预估

- 单树莓派 4B：≈ 50 单/分钟
- 三节点集群：≈ 150 单/分钟
- 建议：村内并发 ≤ 10 人，正常无需扩容
